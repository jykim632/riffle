> 2026-02-10 | 4개 관점 에이전트 팀 리뷰 (UX, 기술 아키텍처, Devil's Advocate, PM)

## 현재 대시보드 구조

### 메인 대시보드 (`/dashboard`)
- **상단**: 스티키 헤더 + 경제지표 스크롤 티커 (USD/KRW, KOSPI, 기준금리, CPI)
- **본문 데스크톱**: 2컬럼 그리드 `[2fr 3fr]`
  - 좌: WeekOverview (주차 정보 + 내 제출 상태 + 전체 제출 현황)
  - 우: CurrentWeekSummaries (이번 주 최근 요약 10개 미리보기)
- **하단**: SeasonBanner (시즌 진행률 + 참여 멤버 목록)
- **모바일**: 수직 스택

### 경제지표 페이지 (`/indicators`)
- 주차 선택 드롭다운
- 12개 지표 그리드 (모바일 2열, sm 3열, lg 4열)
- 각 카드: 지표명 + 현재값 + 전주 대비 변동% + 8주 스파크라인

### 데이터 파이프라인
- 한국은행 ECOS API → Vercel Cron (일일) → economic_indicators 테이블
- 12개 지표: 환율(2), 금리(2), 채권(1), 주가(2), 물가(1), 성장(1), 고용(1), 무역(1), 원자재(1)

## 관점별 상세 분석

### UX 관점

#### 정보 계층
- 경제지표 티커가 최상위를 차지하는 것은 우선순위 역전
- WeekOverview가 좁은 쪽(2fr)에 배치되어 시각적 비중 낮음
- SeasonBanner 하단 배치는 적절하나, 시즌 90%+ 시 상단 노출 필요

#### 사용자 동선
- 요약 제출까지의 CTA가 약함 → 미제출 시 Primary 버튼, 제출 시 Secondary 링크 필요
- 대시보드 티커와 /indicators 페이지의 관계 모호
- 지표 확인 후 "요약 쓰러 가기"로의 브리지 없음

#### 인터랙션 문제
- 스크롤 티커의 인터랙션 모호성 (클릭 가능? 멈출 수 있나?)
- 전주 대비 변동%의 색상 코딩 오해 가능 (CPI 상승 ≠ 좋음, KOSPI 상승 ≠ 나쁨)
- 12개 지표 동일 크기로 나열 → 우선순위 부재

#### 모바일 UX
- 자동 스크롤 티커가 모바일에서 읽기 불가능 + 페이지 스크롤과 충돌
- 10개 요약 카드 수직 스택 → 스크롤 피로
- 경제지표 2열 그리드에서 스파크라인 가독성 의문

#### Quick Wins
1. WeekOverview "내 제출 상태"를 상단 full-width 배너로 분리
2. 티커를 정적 표시로 변경 (자동 스크롤 제거)
3. 모바일 요약 미리보기 10개 → 3~5개 + "전체 보기"
4. 경제지표 카드에 호버 툴팁 (스파크라인 주차별 값)
5. 빈 상태 메시지를 행동 유도 문구로 개선

### 기술 아키텍처 관점

#### 데이터 흐름 효율성
- 레이아웃과 페이지의 워터폴 가능성 확인 필요
- 경제지표 페이지의 3중 쿼리(`getIndicatorsByWeek` + `getPreviousIndicators` + `getIndicatorsHistoryBatch`)가 `Promise.all`로 병렬화되어 있는지 확인 필요
- `getIndicatorsHistoryBatch`가 실제 단일 쿼리인지 확인 (N+1 방지)

#### 캐싱 전략 (현재 부재)

| 대상 | 권장 전략 |
|------|----------|
| 티커 위젯 | `unstable_cache` 또는 `React cache()` + revalidate 3600초 |
| 주차별 지표 | 크론 실행 후 `revalidatePath('/indicators')` on-demand |
| 히스토리 데이터 | 강한 캐싱 (revalidate: 86400+). 과거 데이터는 불변 |

#### 성능 병목

| 병목 | 심각도 |
|------|--------|
| 레이아웃 + 페이지 fetch 워터폴 | 중 |
| 8주 히스토리 배치 쿼리 인덱스 | 중 |
| ECOS API 크론 순차 호출 | 낮 |
| Supabase Cold Start | 낮 |

#### 확장성

| 유형 | 난이도 |
|------|--------|
| 배치 수집형 (뉴스, 추가 지표) | 쉬움 — 기존 크론 패턴 복제 |
| 사용자 생성 (용어사전, 메모) | 쉬움 — 기존 submission 패턴 유사 |
| 정적 참조 (가이드) | 가장 단순 |
| 실시간 (환율 계산기) | 주의 — 기존 배치 패턴과 안 맞음 |
| 실시간 알림/채팅 | 구조 변경 필요 — WebSocket |

#### ECOS API 방어 전략
1. `Promise.allSettled`로 일부 실패해도 나머지 저장
2. UPSERT 패턴으로 실패 시 이전 데이터 유지
3. 크론 실행 결과 로깅 (성공/실패 지표 수)
4. 관리자 대시보드에 "마지막 수집 시각" 표시
5. Zod 파싱으로 형식 변경 조기 감지

#### 기술 체크리스트 (콘텐츠 추가 전 선행)

| 순위 | 항목 |
|------|------|
| 1 | 크론 에러 핸들링 강화 |
| 2 | 경제지표 페이지 쿼리 병렬화 |
| 3 | 캐싱 전략 도입 (특히 티커) |
| 4 | DB 인덱스 확인 (`week_id`, `indicator_code`) |
| 5 | Suspense 경계 정리 |

### Devil's Advocate 관점

#### 현재 기능 중 불필요한 것
- **상단 티커**: 핵심 가치와 무관. 실시간이 아닌데 실시간인 척하는 거짓 신호
- **SeasonBanner 멤버 목록**: 10명짜리 스터디에서 누가 누군지 모르는 사람은 없음. 관리자 페이지 소속

#### 오버엔지니어링 경고
- 12개 경제지표 + 스파크라인: "경제 스터디니까 경제지표가 있어야지"는 논리적 비약. 독서 모임 앱에 서점 재고 현황을 보여주는 수준
- /indicators 페이지: DAU/WAU ≈ 0 가능성. 네이버 증권이 더 낫다
- 10명짜리 앱에 데이터 파이프라인이 존재

#### 유지보수 부담
- ECOS API 장애 전파 위험: 부가 기능 때문에 핵심 기능이 불안정해지는 최악의 시나리오
- 크론잡 모니터링: 10명짜리 앱에서의 운영 부담
- API 키 관리, 요청 제한, 에러 핸들링, 데이터 정합성 — 전부 혼자 관리

#### 기능 추가 반론

| 제안 | 반론 |
|------|------|
| 뉴스 피드 | 외부 의존성 + 저작권 + 관리 비용. 네이버 뉴스가 있다 |
| 환율 계산기 | 네이버에 "환율"이라고 치면 된다 |
| 용어사전 | 초기에 몇 개 넣고 영원히 업데이트 안 됨 |
| AI 요약 | "사람이 직접 쓰는 것"이 앱의 존재 이유. 자기파괴 |

#### 빼야 할 것
1. 상단 티커 → 즉시 제거
2. /indicators 페이지 → 네비에서 숨기기 + 데이터 수집 비활성화
3. 12개 지표 → 최대 3개로 축소 (코스피/환율/금리), 스파크라인 제거
4. SeasonBanner 멤버 목록 → 시즌 진행률만 남기기

#### 핵심 메시지
> "있으면 좋겠다"의 유혹을 이기고 "없어도 된다"를 선택할 수 있어야 1인 개발이 지속 가능하다.

### PM 관점

#### 사용자 핵심 니즈 (우선순위순)
1. **"이번 주 내가 뭘 해야 하지?"** — 제출 독려/상태 확인
2. **"다른 멤버들은 뭘 썼지?"** — 동료 학습(피어 러닝)
3. **"이번 주 경제 뭐가 있었지?"** — 요약 작성에 필요한 맥락 제공

> 이 앱의 본질은 "학습 도구"가 아니라 **"스터디 운영 도구"**

#### 현재 대시보드 가치 평가: 70/100
- 핵심 니즈 1, 2는 잘 커버됨
- 니즈 3(경제 맥락)은 데이터만 있고 해석이 없어서 약함
- "매주 들어올 이유"가 부족

#### 추가 콘텐츠 Impact vs Effort 매트릭스

**Quick Wins (High Impact / Low Effort)**

| 아이디어 | Impact | Effort |
|----------|--------|--------|
| 제출 마감 카운트다운 + 미제출 강조 | 높음 | S |
| 지표 변동 한 줄 해설 (관리자 수동 입력) | 중상 | S |
| 멤버 제출 현황 시각화 개선 | 중상 | S |

**Strategic Bets (High Impact / Medium Effort)**

| 아이디어 | Impact | Effort |
|----------|--------|--------|
| 이번 주 경제 이슈 하이라이트 (수동 큐레이션) | 높음 | M |
| 멤버 활동 통계 (시즌 누적, 스트릭) | 중상 | M |
| 요약 좋아요/이모지 반응 | 중상 | M |

**Later / Avoid**

| 아이디어 | 이유 |
|----------|------|
| AI 요약/분석 | 앱 존재 이유 훼손 |
| 뉴스 피드 | 뉴스 앱이 될 필요 없음 |
| 멤버 랭킹 | 10명 규모 친목 그룹에서 역효과 |

#### MVP 접근 — 적은 노력으로 큰 개선 3가지
1. **제출 마감 카운트다운 + 미제출 강조** (S) → 제출률 직접 향상
2. **지표 한 줄 해설** (S) → 숫자 → 의미 전환
3. **요약 이모지 반응** (M) → 소속감 + 동기부여

#### 릴리스 전략

| Phase | 시기 | 내용 |
|-------|------|------|
| 1 | 이번 주 | 제출 독려 강화 (카운트다운 + 제출 현황 시각화) |
| 2 | 다음 주 | 경제 맥락 제공 (한 줄 해설 + 이슈 하이라이트) |
| 3 | 2~3주 후 | 소셜 레이어 (이모지 반응 + 내 활동 통계) |
| 4 | 한 달+ | 멤버 피드백 보고 결정 |

#### 리텐션 루프

```
[매주 월요일] 새 주차 시작
  → "이번 주 이슈" 카드 노출 (뭘 쓸지 힌트)
  → 제출 마감 카운트다운 시작

[수~목] 제출 시작
  → 미제출 시 urgency 증가
  → 제출 후: 다른 멤버 요약 읽기 유도

[금~일] 다른 멤버 요약 읽기
  → 이모지 반응 → 소속감
  → "이번 주 N명 제출 완료!" 사회적 증거
```

#### 경제 스터디 도메인 특화 제안

| 제안 | 규모 | 설명 |
|------|------|------|
| 이번 주 키워드 위젯 | S | 관리자가 3~5개 경제 키워드 등록, 태그 클라우드 노출 |
| 지표 → 실생활 연결 | S | "환율 1,380원 → 해외직구 비용 약 3% 상승" |
| 손경제 방송 링크 고정 | S | 해당 주 방송 링크/제목을 대시보드에 고정 |

| 용어사전 | 초기에 몇 개 넣고 영원히 업데이트 안 됨 |

## DA의 "빼기" 제안

기능을 더하기 전에 빼기를 먼저 고려:

1. **티커 제거** → 핵심 액션 가시성 확보
2. **12개 지표 → 3~4개 축소** → ECOS 호출 감소 + 장애 표면적 축소
3. **SeasonBanner 멤버 목록 → 관리자 페이지 이동** → 대시보드 간결화
